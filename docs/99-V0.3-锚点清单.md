# V0.3 锚点清单

## 说明

本文从 `docs/03-决策日志.md` 中提取所有显式标注 `V0.3 锚点` 的条目,并纳入 1 条 `V0.2 / V0.3 锚点` 联合标记。

- `触发背景` 保留决策日志原文摘录,不改写原日志。
- `待回答的问题` 按锚点语义整理,供 V0.3 启动时集中讨论。
- `优先级` 为本次整理补充,用于后续排期参考。

| 锚点名 | 触发背景 | 待回答的问题 | 优先级 |
|---|---|---|---|
| 消除“人肉 Orchestrator” | “在 V0.2 文档骨架建设过程中,项目主导者亲身体感到‘人肉 Orchestrator’的痛苦——明明 AI 之间可以传递的信息,必须通过人类复制粘贴中转。每段提示词都需要人类手动从 Claude.ai 复制到 Claude Code 终端,产出再手动复制回来,人在两个 AI 之间充当传递接口。”<br><br>“工具层面消除‘人肉 Orchestrator’需要 IPC 通道(Claude.ai SDK 与 Claude Code SDK 互通),记录为产品后续锚点。” | 1. V0.3 是否要把智能体之间的信息传递从“人肉中转”升级为系统内自动流转?<br>2. 如果引入 IPC 通道,最先需要打通哪些对象:需求、上下文、产出、反馈,还是全链路?<br>3. 自动流转后,用户如何看到传递过程并在冲突、失败、歧义时介入? | P0 |
| 开发期热更新约束再评估 | “V0.2.0 实现阶段需求工作台页面跑出来后,浏览器始终显示旧版占位内容。诊断发现:磁盘文件已是新版,但 Vite 启动时索引的旧版本一直在内存里,后续 Claude Code 通过 Write 工具的多次写入未触发 chokidar 文件监听通知,Vite 不重新 transform 模块,浏览器拿到的是旧 JS。”<br><br>“如果后续切换到其他工具栈或 Claude Code 工具机制变更,本约束可能失效,届时重新评估。” | 1. V0.3 若切换工具栈或工具机制,当前“改完后重启 dev server”的约束是否还成立?<br>2. 新环境下如何验证文件监听、模块重编译、页面刷新链路已经恢复正常?<br>3. 什么条件下可以移除当前兼容性 workaround,回到稳定的热更新体验? | P2 |
| 多项目与协作层级 | “V0.2.0 实现需求工作台时,项目主导者提出‘未来如果是多项目怎么办’的产品问题。当前 V0.2 假设单用户单项目,需求-001、002、003、004 都属于同一个隐含的项目。”<br><br>“V0.2 仍按单用户单项目设计,不引入‘项目’层级。但本观察记下,作为 V0.3 锚点。” | 1. “项目”是否作为新的层级引入(需求 → 任务 → 运行之上加一层)?<br>2. 多项目时,需求工作台是否按项目分组,还是用项目切换器?<br>3. 智能体团队是项目级共享还是需求级独立?<br>4. 多用户场景下,如何展示“别人正在做什么”? | P0 |
