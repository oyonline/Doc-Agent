## PRD

```
# PRD · 新增用户标签功能(需求-001)

## 背景

当前用户主页缺少快速展示用户身份特征的轻量入口,用户在头像旁只能看到用户名。增加自由输入的标签机制可让用户低成本表达个性,同时为管理员提供 UGC 治理工具。

## 目标

- 用户可自由输入并管理自己的标签,标签展示在主页头像下方。
- 管理员可对全平台标签做发布后审核,违规标签可被隐藏并通知用户。
- 数据模型与 API 设计上对未来"标签搜索 / 标签社群"等扩展保持开放。

## 范围

### 包含

- 用户标签 CRUD(增 / 改 / 删 / 查)
- 用户主页标签展示组件(头像下方一行)
- 管理员审核界面(列表 + 单条审核)
- 后端审核 API(批准 / 隐藏)
- 数据库 schema(用户与标签的关联表)

### 不包含

- 标签搜索能力(留 V0.3)
- 标签互动(关注同标签用户、标签社群等,留 V0.3)
- 规则化自动审核(留 V0.3)

## 用户故事

- 作为**普通用户**,我希望能给自己打最多 5 个标签,让我的主页更具个性,以便他人快速理解我。
- 作为**用户主页访问者**,我希望能在头像下方一行内看到该用户的全部标签(不超过 5 个),不需要展开多行或点击。
- 作为**管理员**,我希望能在审核界面快速过滤新提交的标签并做批准 / 隐藏决策,以便控制 UGC 风险。

## 需求级验收标准

1. 用户可在主页编辑组件中添加 / 删除自己的标签,标签数量上限 5 个;超过 5 个时无法新增,组件给出明确提示。
2. 标签长度 ≤ 20 字符,支持 emoji(数据库使用 utf8mb4)。
3. 管理员审核界面可分页查看全平台标签,可对单条标签执行"批准"或"隐藏"操作。
4. 隐藏后的标签不再在用户主页显示,但保留在数据库供后续申诉与统计。
5. 任务-2、3、4、5、6、7 全部任务内验证通过(单元测试 / 代码检查 / 类型检查 / 契约校验),且任务-7 集成测试覆盖前后端联调主路径。
6. 标签内容存储与展示均不允许 HTML 标签透传:服务端入库前转义 + 前端禁用 dangerouslySetInnerHTML 渲染标签。 [本条由协同澄清第 4 轮记录追加 — 架构师识别 XSS 风险,用户当场给出双保险策略,直接进 PRD 验收]
```

## 可行性评估(规划阶段并行产出)

> 本章节为协作模型并行升级后的事实源。R-001 XSS 风险在协同澄清第 4 轮由架构师识别,用户当场给出双保险策略;规划阶段架构师独立产出可行性评估时,R-001 直接定为中风险,缓解建议完整落地。**不再有"驳回 → 二次澄清 → 降级"动线。**

### 5.1 风险清单(5 条 · 0 高 + 4 中 + 1 低)

架构师智能体在规划阶段(14:10-14:14)产出可行性评估,识别 5 条风险。R-001 经协同澄清第 4 轮已敲定缓解策略,直接定为中。

| 编号 | 等级 | 描述 | 影响范围 | 缓解建议 |
|---|---|---|---|---|
| R-001 | 中 | 标签注入风险:用户输入若未严格校验可能 XSS | 用户主页展示;管理员审核界面;移动端 webview 内嵌主页 | 服务端 escape + 前端 React 自动转义 + 禁用 dangerouslySetInnerHTML 双保险;任务-4、5 验收标准追加"ESLint 规则不允许 dangerouslySetInnerHTML"。**缓解策略源自协同澄清第 4 轮(D-1)** |
| R-002 | 中 | 标签搜索性能:大用户量下 LIKE 查询慢(虽然 PRD 约束"暂不参与搜索",但管理员审核界面已经隐含搜索) | 审核接口;后续标签搜索扩展 | 加 (status, created_at) 复合索引;后续可扩展全文搜索 |
| R-003 | 中 | 审核滞后导致违规暴露:发布后审核窗口期可能展示违规标签 | UGC 风险,平台合规 | 违规模式词典自动隐藏 + 高频提交用户白名单 |
| R-004 | 中 | 数据库迁移影响:user 表加新关联表,在线迁移需保证用户读路径不卡顿 | 用户主页读路径(高 QPS) | 索引提前创建 + 灰度发布 + 迁移可逆校验。**与协同澄清第 4 轮架构师建议一致** |
| R-005 | 低 | emoji 标签存储:必须 utf8mb4 字符集,否则 emoji 入库后变 ? | DB 字段 | 在 schema 设计阶段确认 utf8mb4,任务-1 验收。**与协同澄清第 4 轮架构师建议一致** |

#### 评估结论

最终风险分布:**0 高 + 4 中 + 1 低**,满足进入闸门 2 条件。

### 5.2 实现方向建议

> 字段级细节(utf8mb4、escape、ENUM 取值等)已在协同澄清记录中敲定,本节不重复。本节聚焦架构师独立产出的技术栈选型与实现策略。

- **后端技术栈**:Node.js + TypeScript + Prisma ORM(沿用项目现有栈)。详见澄清记录第 1 轮(关联表存储)、第 2 轮(状态枚举建模)、第 3 轮(utf8mb4)。
- **前端实现策略**:React + TypeScript;主页标签编辑组件复用项目现有 ChipInput 组件,5 上限通过 props 控制(对应 D-10);审核界面新建 `AdminTagReview` 路由组件,使用项目现有 Table 模块;ESLint 规则强约束禁用 dangerouslySetInnerHTML(兑现澄清第 4 轮的双保险策略)。
- **测试栈与策略**:单元测试 vitest 沿用;**集成测试任务-7 用 supertest** 跑前后端联调;**端到端测试任务-6 用 playwright**,主路径"普通用户登录 → 编辑标签 → 管理员登录 → 审核 → 主页验证显示"。
- **迁移策略**:索引提前创建 + 灰度发布 + 迁移可逆校验(对应 R-004)。详见澄清记录第 4 轮架构师建议。

### 5.3 关键技术决策记录(架构师层面)

1. **状态枚举命名**:`pending / approved / hidden`(不用 active / inactive,保留"被隐藏"语义)。
2. **关联表 vs 用户字段**:用关联表(`user_tags`),不在 user 表加 JSON 字段。理由:审核与统计场景需独立索引。
3. **utf8mb4 范围**:标签字段使用 utf8mb4,user 表保持现有 utf8(范围最小化变更)。
4. **XSS 双保险策略**:服务端 escape + 前端禁用 dangerouslySetInnerHTML(源自**协同澄清第 4 轮**架构师识别 + 用户决策,落到任务-2 / 任务-4 / 任务-5 验收)。

+ import { tagInputSchema } from './tags.schema'

+
+ export const tagsRouter = Router()
+
+ // GET /api/tags — 当前用户的标签列表(游标分页)
+ tagsRouter.get('/', async (req, res) => {
+   const cursor = req.query.cursor ? Number(req.query.cursor) : undefined
+   const tags = await prisma.userTag.findMany({
+     where: { userId: req.user.id, status: { not: 'hidden' } },
+     take: 20,
+     ...(cursor ? { skip: 1, cursor: { id: cursor } } : {}),
+     orderBy: { createdAt: 'desc' },
+   })
+   res.json({ tags, nextCursor: tags.at(-1)?.id ?? null })
+ })
+
+ // POST /api/tags — 新建标签(默认 pending)
+ tagsRouter.post('/', async (req, res) => {
+   const parsed = tagInputSchema.parse(req.body)
+   const safeName = escape(parsed.tagName)
+   // 同用户重复名静默忽略
+   const existing = await prisma.userTag.findFirst({
+     where: { userId: req.user.id, tagName: safeName },
+   })
+   if (existing) return res.status(200).json({ tag: existing })
+   const created = await prisma.userTag.create({
+     data: {
+       userId: req.user.id,
+       tagName: safeName,
+       status: 'pending',
+     },
+   })
+   res.status(201).json({ tag: created })
+ })

### 11.4 异常路径样例数据

(对应 `docs/04 "## 异常路径"` 6 条路径,本节填充具体样例数据)

#### 路径 1:任务执行失败(任务-2 执行中失败)

- 错误信息:"无法连接到测试数据库 · 迁移未应用 · 详见日志 14:23:11"
- 任务状态:执行中 + 执行失败
- 智能体状态:后端智能体 工作中→空闲;项目管理智能体 不调度下游(任务-7、4、5 等待)
- 恢复路径:用户点击"重试"→ 任务-2 重新执行

#### 路径 2:任务内验证未通过(任务-3 验证未通过)

- 验证失败项:"标签审核状态枚举值 typo · `aprrove` 应为 `approve`,导致 POST /api/admin/tags/:id/approve 返回 500"
- 任务状态:验证中 + 验证未通过 + 需要人工处理
- 恢复路径:用户点击"重试"→ 任务-3 回到执行中

#### 路径 3:用户停止任务(用户在任务-2 执行中点击"停止")

- 停止时进度:已完成 GET、POST,PUT 写到一半 · 已运行 3 分 47 秒
- 任务状态:执行中 + 已停止
- 恢复路径:用户点击"重试"→ 任务-2 重新执行(新建一次运行,不带原会话状态)

#### 路径 4:用户拒绝单任务审查包(用户拒绝任务-1)

- 反馈文字:"schema 字段命名 `user_tag` 应改为 `user_tags`,与项目命名规范一致;另:索引建议改为复合索引 `(status, created_at)`,而不是单列 status 索引"
- 任务状态:待审查 → 待处理 + 已拒绝
- 反馈进入下次运行上下文(任务详情页"上下文"中的"上一次反馈"字段填入此文字)
- 下游影响:任务-2、3 仍待处理(因为任务-1 没"已完成")
- 恢复路径:任务-1 重新执行,带反馈进入新运行

#### 路径 5:任务卡住(任务-3 长时间无更新)

- 卡住时长:无更新已 8 分 12 秒
- 任务状态:执行中 + 卡住
- 恢复路径:用户介入,选择"重试"或"停止"

#### 路径 6:用户在需求级"部分接受"

- 用户接受任务-1、2、4、5、6,拒绝任务-3、7
- 拒绝理由:"任务-3 审核 API 默认状态错误(对应跨任务严重问题 #1);任务-7 集成测试需重跑因为 #3 修复后契约可能变化"

- 任务-3、7 状态:已完成 → 待处理 + 已拒绝
