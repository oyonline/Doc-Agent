# V0.2 · 任务调度规范

## 文档目的

本文档定义 V0.2 的任务调度时序、并行边界、契约依赖处理、异常路径,以及 V0.2.0 / V0.2.1 / V0.2.2 三个批次的调度差异。后续代码 Agent 在实现"任务图节点状态推进""任务卡状态变化""智能体状态联动""事件流推送"等所有调度相关行为时,以本文档为指令源。

本文档不重复定义状态机(见 `docs/03-V0.2-状态机与跳转矩阵.md`)、不定义页面布局(见 `docs/08-V0.2-页面任务书.md`)、不定义具体样例数据值(见 `docs/06-V0.2-样例数据与事件文案.md`)。

## 调度核心原则

`docs/02-V0.2-八智能体职责说明.md` 在 `### 4.4 项目管理智能体` 章节已定义 5 条调度规则(实线前置依赖满足才可调度、可调度任务同时入队、智能体可同时处理多任务、自动放行 + 人工介入节点暂停、单任务审查包始终生成不阻塞),本文档不重复展开。

V0.2 调度的核心原则在 5 条规则之上追加:

1. 依赖驱动,不靠时间表。任务何时调度只看依赖是否满足,不预设"几点跑哪个"。
2. 同时可调度任务必须同时分发,不允许人为排队。这是"真并行"的硬要求。
3. 自动放行是默认,人工介入是例外。只在 docs/00 锁定的 4 个介入节点暂停,其他节点自动推进。
4. 单任务审查包始终生成,不阻塞自动放行。它是"可主动查看"的资产,不是"必须等待"的关卡。
5. 异常路径不省略,不简化。失败、拒绝、停止、卡住都有明确的回退路径。

## 并行的三种类型

V0.2 演示三种不同的并行,分清楚才能精确实现:

### 类型一:模块独立并行

两个任务之间无任何共享数据/接口/文件,可完全独立同时进行。

样例:任务-4(用户主页标签编辑组件) 与 任务-5(管理员审核界面)。

- 都是前端任务,改的文件不重叠
- 调用的后端 API 不同(任务-2 vs 任务-3)
- 不需要任何额外协调

调度规则:依赖前置满足后立即同时分发。

### 类型二:契约先行的并行

两个任务有数据契约关系,但只要契约先固定下来,可基于契约同时开发。

样例:任务-2(标签 CRUD API) 与 任务-4(用户主页标签编辑组件)。

- 前端要调后端 API,严格说是"后端做完前端才能开始"的串行
- 但只要 API 契约(请求路径、参数、返回结构)在拆解阶段就固定,前端可基于 mock 接口先开发

V0.2.0 阶段:不演示契约先行的并行,任务-2、3 完成后直接放行任务-4、5。

V0.2.1 阶段:任务图新增虚线契约依赖,但实际调度仍按"任务-7 完成后放行任务-4、5"。虚线只表达"这两个任务通过契约关联,代表了真实工程里的契约先行可能性,但 V0.2 不演示真正的契约先行调度"。

### 类型三:完成时间错位的并行

下游任务的"开始"可早于上游任务的"完成",只要上游到了某个里程碑。

样例:任务-6(端到端测试) 与 任务-4(前端组件)在真实工程中可错位。

V0.2 不演示这种并行(留 V0.3 锚点)。

## 串行的三种类型

### 类型一:强数据依赖

样例:任务-1(数据库 schema) → 任务-2(标签 CRUD API)。

理由:没有 schema 就没有表,API 没法实际跑通。

调度规则:任务-2 在任务-1 状态变成"已完成"前,绝对不开始。

### 类型二:契约依赖(契约还未产出)

样例:任务-1(数据库 schema) → 任务-3(标签审核 API)。

理由:虽然任务-3 是另一个 API,但要用任务-1 定义的 schema。

调度规则:同强数据依赖,等任务-1 完成。

### 类型三:汇聚型串行(全部完成才能开始)

样例:任务-7(后端集成测试) 依赖任务-2、3 全部完成。
样例:任务-6(端到端测试) 依赖任务-1、2、3、4、5、7 全部完成。

调度规则:汇聚节点等待所有前置任务都进入"已完成"状态。

## V0.2.0 阶段调度时序差异

V0.2.0 阶段任务-7 是占位,不参与调度。完整时序简化为:

```
T0   任务-1 启动
T1   任务-1 完成 → 待审查(人工介入)
     ↓ 用户接受
T1+  任务-2、3 启动(并行)
T2   任务-2 完成,任务-3 仍在跑(参差状态)
T2+  任务-3 完成 → 任务-2、3 自动放行 → 任务-4、5 启动(并行,跳过任务-7)
T3   任务-4、5 都完成 → 任务-6 启动
T4   任务-6 完成 → 待审查(人工介入)
     ↓ 用户接受
T5   需求级审查包生成 → 用户最终决策
```

V0.2.0 与 V0.2.1 的关键差异:

- 没有 T2+/T3 之间的"任务-7 闸门"
- 任务-2、3 完成后直接放行任务-4、5,不经集成测试
- 人工介入次数减少 1 次(没有任务-7 后的介入)

## 异常路径

V0.2 必须演示但不主动跑通的异常路径(用户可点击"演示异常"切换到这些状态):

### 异常路径 1:任务执行失败

样例:任务-2 执行中失败。

任务状态变化:

- 任务-2:执行中 → 执行中 + 执行失败 标签

智能体状态:

- 后端智能体:工作中 → 空闲
- 项目管理智能体:不调度下游(任务-7、4、5 等待)

恢复路径:用户点击"重试" → 任务-2 重新执行。

### 异常路径 2:任务内验证未通过

样例:任务-3 验证中失败。

任务状态变化:

- 任务-3:验证中 → 验证中 + 验证未通过 标签

恢复路径:用户点击"重试" → 任务-3 回到执行中。

### 异常路径 3:用户停止任务

样例:用户在任务-2 执行中点击"停止"。

任务状态变化:

- 任务-2:执行中 → 执行中 + 已停止 标签

恢复路径:用户点击"重试" → 任务-2 重新执行。

### 异常路径 4:用户拒绝单任务审查包

样例:用户拒绝任务-1 审查包并填写反馈。

任务状态变化:

- 任务-1:待审查 → 待处理 + 已拒绝 标签
- 反馈进入下次运行上下文

下游影响:任务-2、3 仍待处理(因为任务-1 没"已完成")。

恢复路径:任务-1 重新执行,带反馈进入新运行。

### 异常路径 5:任务卡住(长时间无推进)

样例:任务-3 在执行中长时间无更新。

任务状态变化:

- 任务-3:执行中 → 执行中 + 卡住 标签

恢复路径:用户介入,选择重试或停止。

### 异常路径 6:用户在需求级"部分接受"

样例:用户接受任务-1、2、4、5、6,拒绝任务-3、7。

需求状态变化:

- 需求-001:待审查 → 执行中 + 部分拒绝 标签

任务状态变化:

- 任务-3、7:已完成 → 待处理 + 已拒绝 标签
- 任务-1、2、4、5、6:已完成保持

调度恢复:项目管理智能体重新分发任务-3,任务-7 等任务-3 完成后跑。

## 严禁事项

- 严禁让任务图节点在 T1 → T1+ 之间无视觉过渡(必须能看到节点状态颜色变化)。
- 严禁让"参差状态"(T2)被简化为"任务-2、3 同时完成"。这是真并行的核心证据。
- 严禁让自动放行机制跳过单任务审查包的生成(审查包始终生成,只是不阻塞)。
- 严禁在 V0.2.0 阶段让任务-7 进入任何调度状态(它是占位)。
- 严禁让异常路径只演示"失败 → 修复",必须包含"失败 → 状态外化 → 用户决策 → 恢复"完整路径。
- 严禁绕过项目管理智能体让执行智能体直接接收新任务(所有调度都通过项目管理智能体中介)。
