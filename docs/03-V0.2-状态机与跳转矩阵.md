# V0.2 · 状态机与跳转矩阵

## 文档目的

本文档定义 V0.2 中需求、任务、运行三层粒度的状态机,以及智能体状态机、页面跳转矩阵、按钮触发后的状态变化。它是 V0.2 的"行为契约"——后续代码 Agent 实现页面交互时以本文档为指令源:按状态机推动数据,按跳转矩阵连接页面,按按钮触发矩阵规定每个动作的副作用。

本文档不定义页面布局、不定义样例数据具体值、不定义 UI 文案具体文字。这三类内容由 `docs/06-V0.2-样例数据与事件文案.md` 与 `docs/08-V0.2-页面任务书.md` 等后续文档承担。

## 三层粒度概览

简要回顾(引用 `docs/00-V0.2-范围对齐.md` "## 三层粒度",不重写):需求 / 任务 / 运行三层独立运转,但通过包含关系联动。

附一张关系图(纯文字版):

```
需求(Requirement)
  ├─ 任务-1(Task)
  │     └─ 运行-1(Run)
  │     └─ 运行-2(Run,如重试)
  ├─ 任务-2(Task)
  │     └─ 运行-1
  └─ ...
```

说明:一个需求包含多个任务,一个任务可包含多次运行(重试或失败重做)。

## 需求状态机

### 主状态(6 个,严格,含 1 个异常终态)

| 主状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| 待澄清 | 需求刚提交,等待产品智能体启动协同澄清;或被异常标签"放弃/拆分"叠加,等待用户决策 | 用户提交需求 | 产品智能体启动协同澄清(第 1 轮开始) |
| 规划中 | 含两个子阶段:**澄清子阶段**(产品 + 架构师协同澄清,4 轮)与**产出子阶段**(三产出并行 + 闸门 2 review) | 产品智能体启动协同澄清第 1 轮 | 闸门 2 用户点击"启动执行"(进入"执行中");或闸门 2 用户点击"重新澄清"(回到"待澄清") |
| 执行中 | 任务图已就绪,正在执行 | 闸门 2 启动执行 | 所有任务都进入"已完成"或"已拒绝"状态 |
| 待审查 | 所有任务执行完,需求级审查包已生成,等待人工最终决策 | 所有任务都进入"已完成"且需求级审查包已生成 | 用户在需求级审查包页做决策 |
| 已完成 | 用户接受需求级审查包 | 用户点击"接受需求结果" | 终态,不再退出 |
| **已放弃** | **异常终态**,带子语义"放弃 / 拆分"。原需求作废 | 用户在"待澄清 + 放弃/拆分(异常标签)"状态下选择"放弃"或"拆分" | 终态,不再退出 |

**子阶段说明**(规划中主状态内部):

- **澄清子阶段**:协同澄清进行中,1-4 轮。第 4 轮结束后,产品智能体宣布对齐 → 整合 → 展示澄清结论摘要 → 等待用户在闸门 1 点击"冻结需求,开始规划"。期间右侧状态栏与需求详情页顶部进度指示展示"澄清"段高亮。
- **产出子阶段**:闸门 1 后启动,三产出并行(PRD ‖ 可行性评估 ‖ 任务图)。三产出完成后展示规划 review 视图,等待用户在闸门 2 决策。期间进度指示展示"三产出"段或"review"段高亮。

子阶段在 UI 上由轻量进度指示呈现(澄清 ▶ 三产出 ▶ review),状态色与状态文案保持单一"规划中"。详见 `docs/00 §三层粒度` 与 `docs/08 §页面通用规范`。

### 异常标签(可叠加在主状态上,不替代主状态)

| 异常标签 | 适用场景 | 处理方式 |
|---|---|---|
| 放弃/拆分 | **系统自动施加**:协同澄清 ≥ 6 轮仍未对齐(产品与架构师都未给出"已对齐"信号)。提示用户做决策 | 叠加在"待澄清"主状态上。用户在三选项中选择:"继续尝试澄清"(移除标签,保持"待澄清",澄清继续)/"放弃"(主状态进入终态"已放弃(放弃)")/"拆分"(主状态进入终态"已放弃(拆分)",新需求若干各自从"待澄清"开始) |
| 部分拒绝 | 用户在需求级审查包选择"部分接受",拒绝部分任务 | 主状态从"待审查"回到"执行中",被拒任务回任务池重做 |
| 全量拒绝并反馈 | 用户在需求级审查包选择"拒绝并反馈" | 主状态从"待审查"回到"规划中"(产出子阶段)或"执行中",根据反馈决定 |

### 状态转换图(文字版,并行模型)

```
待澄清 ─────────┐
  │             │ 系统自动施加
  │ 产品启动     │ 放弃/拆分(异常标签)
  │ 协同澄清     │  · 继续尝试澄清 → 移除标签,留在"待澄清"
  │             │  · 放弃 → 已放弃(放弃)〔终态〕
  │             │  · 拆分 → 已放弃(拆分)〔终态〕+ 新建若干"待澄清"
  ▼             ▲
规划中(澄清子阶段)
  │ 第 4 轮完成 + 整合澄清结论摘要
  │
  ▼ 闸门 1 · 用户冻结需求,开始规划
规划中(产出子阶段)
  │ 三产出完成 + 进入闸门 2 review
  │
  ├─ 闸门 2 · 重新澄清 ─→ 待澄清(所有产出作废)
  │
  ▼ 闸门 2 · 启动执行
执行中 ←─────────┐
  │              │ 部分拒绝 / 全量拒绝并反馈
  │ 所有任务完成  │
  ▼              │
待审查 ──────────┘
  │ 接受需求结果
  ▼
已完成(终态)

异常终态:已放弃(放弃 / 拆分)— 由协同澄清"放弃/拆分"异常标签上的用户决策推入。
```

**注**:闸门 2 的"小修补"(对 PRD / 可行性评估 / 任务图任意一段)不改变主状态,只让对应 Agent 重新工作中。详见按钮触发矩阵。

## 任务状态机

### 主状态(5 个,严格,V0.1.1 资产保留)

| 主状态 | 含义 | 进入条件 | 退出条件 |
|---|---|---|---|
| 待处理 | 任务已加入任务池,等待依赖前置完成 | 任务加入任务池 | 所有依赖前置都"已完成" |
| 执行中 | 智能体正在执行任务 | 项目管理智能体分发任务 + 智能体接收 | 智能体完成代码改动 |
| 验证中 | 任务内验证正在跑(单元测试、代码检查等) | 智能体完成代码改动 | 任务内验证全部通过或失败 |
| 待审查 | 任务内验证通过,单任务审查包已生成 | 任务内验证全部通过 + 审查包生成 | 调度推进 / 用户人工接受 / 用户拒绝 |
| 已完成 | 调度推进或用户人工接受 | 自动放行 / 用户接受单任务审查包 | 终态,但仍可被需求级"部分拒绝"打回到"待处理" |

### 异常标签

| 异常标签 | 适用场景 |
|---|---|
| 已停止 | 用户在任务详情页点击"停止" |
| 执行失败 | 智能体执行阶段失败 |
| 验证未通过 | 任务内验证至少一项未通过 |
| 已拒绝 | 用户在单任务审查包选择"拒绝并反馈" |
| 卡住 | 长时间无推进的静态提示 |
| 需要人工处理 | 处于人工介入节点(任务-1 后、任务-7 后、任务-6 后)等待人工 |

### 状态转换图(文字版)

```
待处理
  │ 依赖满足 + 项目管理智能体分发
  ▼
执行中 ── 失败 ──→ 执行中 + 执行失败 → (重试)→ 执行中
  │ 代码改动完成
  ▼
验证中 ── 验证失败 ──→ 验证中 + 验证未通过 → (重试)→ 执行中
  │ 验证全部通过
  ▼
待审查
  │
  ├─ 自动放行(非人工介入节点)─→ 已完成
  ├─ 用户接受 ─→ 已完成
  └─ 用户拒绝并反馈 ─→ 待处理 + 已拒绝(回任务池重做)
```

## 运行状态机

### 主状态(5 个,V0.1.1 资产保留)

| 主状态 | 含义 |
|---|---|
| 未开始 | 运行记录已准备但未启动 |
| 进行中 | 当前运行正在执行 |
| 已完成 | 当前运行完成 |
| 已失败 | 当前运行失败 |
| 已停止 | 当前运行被用户停止 |

### V0.2.2 扩展:会话分段

V0.2.2 启用 Context 管理深度后,运行内部增加"会话分段"概念:

- 一次运行可包含多段会话(第 1 段、第 2 段、...)
- 每段会话有自己的开始时间、结束时间、归档原因(上下文接近上限 / 任务完成 / 用户停止)
- 段与段之间通过"接续摘要"承接,接续摘要必须包含决策记录

会话分段不改变运行的主状态,只在任务详情页"决策与上下文"Tab 展示分段记录。

## 智能体状态机

每个智能体只有两种主状态:**空闲 / 工作中**。

工作中可叠加细分文案,详见 `docs/02-V0.2-八智能体职责说明.md` "状态展示规范"章节,本章不重复。

V0.2.2 启用后,执行阶段智能体(前端、后端、测试)的状态额外展示当前会话信息(第几段、是否接续),不显示百分比。

### 任务表示形态说明

V0.2 中"任务"在不同页面有两种 UI 表示形态:

- **任务图节点**:出现在需求详情页,展示任务在依赖图中的位置,通过连线表达依赖关系(实线 / 虚线)。任务图节点是依赖图的组成部分,不能脱离上下文单独出现。
- **任务卡**:出现在任务板,以列表形态展示单个任务的状态。任务卡是孤立单元,不展示依赖关系。

两种形态点击后都跳转到任务详情页,但实现上必须分别构建,不允许把任务图节点当作"任务卡的迷你版"。

## 按钮触发矩阵

按钮维度的精确指令(给代码 Agent 实现用)。

| 按钮 | 所在页面 | 二次确认 | 是否需要反馈 | 主状态变化 | 异常标签变化 | 涉及智能体状态 |
|---|---|---|---|---|---|---|
| 创建需求 | 顶栏 | 否 | 否 | 不变 | 不变 | 不变 |
| 提交需求(需求工作台新建弹层) | 需求工作台 | 否 | 否 | 新建需求,主状态 = 待澄清 | 不变 | 产品 + 架构师同时准备启动 |
| 提交回答(协同澄清) | 需求详情页 | 否 | 否 | 规划中(澄清子阶段)· 第 N 轮 → 第 N+1 轮 | 不变 | 产品(主持澄清)+ 架构师(协同澄清)同时工作中 |
| 冻结需求,开始规划(闸门 1) | 需求详情页 | 否 | 否 | 待澄清 / 规划中(澄清子阶段) → 规划中(产出子阶段) | 不变 | 产品智能体空闲(整合完成);三智能体并行启动:产品(撰写 PRD)、架构师(可行性评估)、任务拆解(等待规划稳定 → 拆解中) |

## V0.2.0 / V0.2.1 / V0.2.2 批次差异说明

每个状态、每个按钮在不同批次的可用性差异:

### V0.2.0 阶段

- 任务-7 在任务图中作为"样例占位",不进入任务状态机(永远不会变成"执行中"或"已完成")。
- 闸门完整 5 个:闸门 1(冻结需求)、闸门 2(启动执行)、闸门 3(任务-1 后)、闸门 4(任务-6 后)、闸门 5(需求级审查)。
- 任务图节点只显示实线箭头。
- 智能体状态不显示会话分段信息。
- 任务详情页"决策与上下文"Tab 只显示初始上下文,不显示决策日志和会话切换记录。

### V0.2.1 阶段(在 V0.2.0 基础上变化)

- 任务-7 激活,加入任务状态机和调度链路。
- 闸门完整 6 个:在 V0.2.0 基础上插入闸门 4 · 任务-7 后集成闸门;原任务-6 后端到端闸门顺位为闸门 5,需求级审查顺位为闸门 6。
- 任务图节点新增虚线契约依赖(任务-2 与任务-4、任务-3 与任务-5)。
- 调度链路调整:任务-2、3 完成后必须经任务-7 闸门才放行任务-4、5。

### V0.2.2 阶段(在 V0.2.1 基础上变化)

- 任务详情页"决策与上下文"Tab 完整启用,显示决策日志、会话切换记录。
- 智能体状态行下方附加当前会话信息("第 N 段会话 · 接续自第 M 段")。
- 事件流出现"上下文已归档,启动新会话继续"事件。

## 严禁事项

- 严禁让任务的"已完成"状态被理解为"终态不可改变",它仍可被需求级"部分拒绝"打回到"待处理"。
- 严禁让需求状态、任务状态、运行状态使用相同名词时混用(例如"待审查"在需求级和任务级语义不同,必须始终标注层级)。
- 严禁让异常标签替代主状态,异常标签必须叠加。"已放弃"是主状态而非异常标签,与本条规则无关。
- 严禁让 V0.2.0 阶段的实现把任务-7 当作真实任务(它只是占位)。
- 严禁绕过"按钮触发矩阵"自行添加未列出的状态变化。
- 严禁让闸门(闸门 1 至闸门 5/6)被自动跳过。
- 严禁让"已驳回"以任何形式重新进入需求状态机(包括以异常标签 / 主状态 / 子状态 / 别名等任何形式)。
- 严禁让"放弃/拆分"以异常标签形式留存——它已升级为主状态"已放弃",在异常标签表中只承担"系统自动施加的提示标签"角色,不承担终态语义。
- 严禁让闸门 2 的"小修补"改变主状态(主状态保持"规划中(产出子阶段)",仅对应 Agent 重新工作中)。
- 严禁让闸门 1 的子阶段切换被自动触发(必须由用户点击"冻结需求,开始规划")。
