# V0.2 · Context 管理规范

## 文档目的

本文档定义 V0.2.2 启用的智能体会话管理规则,包括会话切换触发条件、归档与接续机制、UI 呈现规范、异常处理。后续代码 Agent 实现会话切换演示时,以本文档为指令源。

本文档不定义状态机(见 `docs/03-V0.2-状态机与跳转矩阵.md`)、不定义页面布局(见 `docs/08-V0.2-页面任务书.md`)、不定义具体样例数据值(见 `docs/06-V0.2-样例数据与事件文案.md`)。本文档不实现真实 context 监控,V0.2 全程使用预设的样例切换点演示。

## 为什么需要会话管理

真实工程中,智能体执行长任务时,上下文长度会逐渐接近模型可承载的上限,此时必须主动归档当前会话并新开会话才能保证任务连续性。这本身不可避免。

但如果切换不当,任务质量就会下降——智能体可能忘记之前的关键决策、重做已完成的步骤、或在新旧会话之间产生不一致的输出。这是会话切换最关键的风险。

V0.2.2 把"会话切换"作为可见信号展示给用户,目的是让人类知道系统在主动管理 AI 的上下文边界,而不是把这件事藏在黑盒里。本文档不展开真实 token 监控、压缩策略等技术实现细节。

## 会话切换触发规则

V0.2.2 使用预设样例切换点演示,**不监控真实 token 数**。

### 触发类型(三种)

#### 类型一:执行步骤数触发

样例规则:智能体在单次会话内执行超过 10 个关键动作(读文件、写文件、跑测试等),触发归档。

理由:动作数比 token 数更能预测 context 增长趋势。

V0.2 演示:在样例数据中预设特定任务的特定动作数到达阈值时触发切换。

#### 类型二:关键里程碑触发

样例规则:智能体完成一个内部里程碑(如"schema 设计完成"、"主要 API 写完"),主动归档以保留干净状态供下一里程碑接续。

理由:在干净的里程碑边界归档,接续摘要更容易组织,质量更可控。

V0.2 演示:在样例数据中预设里程碑切换点。

#### 类型三:用户主动触发

样例规则:用户在任务详情页主动点击"归档当前会话"按钮,智能体立即归档新开。

理由:用户可能在 review 中发现智能体走偏,希望"重启"上下文但保留任务进度。

V0.2 演示:任务详情页的会话区域提供该按钮。

### V0.2.2 演示的样例切换分布

V0.2.2 阶段在样例需求"新增用户标签功能"中,预设以下任务有会话切换演示:

```
任务-2(标签 CRUD API):3 段会话
  - 第 1 段:读 schema、设计接口、写 GET/POST(动作数触发归档)
  - 第 2 段:写 PUT/DELETE、写参数校验(动作数触发归档)
  - 第 3 段:跑测试、修 bug、生成审查包(任务完成自然结束)

任务-4(用户主页标签编辑组件):2 段会话
  - 第 1 段:读组件、设计接口、写主体逻辑(动作数触发归档)
  - 第 2 段:补样式、跑测试、生成审查包(任务完成自然结束)

其他任务(任务-1、3、5、6、7):各 1 段会话(单段完成,不演示切换)
```

具体动作数和里程碑细节见 `docs/06-V0.2-样例数据与事件文案.md`。

## 归档时必须保留的内容

会话归档时,智能体必须产出"接续摘要 + 自检结果",作为下一段会话的输入。

### 接续摘要必须包含

- **已完成**:本段会话已完成的具体动作清单(精简,只列结果不列过程)。
- **决策**:本段会话做出的关键决策(必须包含理由,不只是结论)。
- **未决**:本段会话遇到但未解决的问题(留给下一段或用户)。
- **下一步**:下一段会话应当继续做的事。
- **关键文件**:本段会话涉及的文件清单(供下一段会话快速定位)。

### 自检结果必须包含

- **未完成事项检查**:确认本段会话的"未完成"已经全部记录。
- **持久化检查**:确认状态、决策、关键发现已写入接续摘要。
- **接续准备度**:确认下一段会话能基于摘要继续工作,不丢失上下文。

### 严禁

- 严禁让接续摘要只记录"做了什么"而不记录"为什么这么做"。
- 严禁让接续摘要超过 500 字(超出说明摘要不够精简)。
- 严禁丢弃用户在本段会话给出的反馈或指令。

## 接续会话的启动规则

新会话启动时,**只读以下内容**(避免重新带入历史上下文):

1. 任务卡(任务名、目标、依赖、验收标准)
2. 接续摘要(上一段会话产出)
3. 必要的代码切片(从接续摘要的"关键文件"清单按需读取)

新会话**不读以下内容**:

1. 上一段会话的完整对话历史
2. 上一段会话读取过的所有文件原文
3. 上一段会话的完整工具调用记录

理由:context 是稀缺资源,只读决策不读过程,质量和效率都更高。

## UI 呈现规范

### 任务详情页"决策与上下文"Tab(V0.1.1"上下文"Tab 升级)

V0.2.2 启用后,Tab 内容分三块从上到下:

#### 块 1:初始上下文

- 角色、任务、验收标准、项目规则、工作区、需要运行的验证、上一次反馈、安全模式
- (V0.1.1 沿用结构,字段不变)

#### 块 2:决策日志

任务执行过程中产生的关键决策,按时间顺序展示。

每条决策包含:

- 时间(HH:MM)
- 决策内容(一句话)
- 理由(一段话)
- 来自哪个会话(第 N 段)

#### 块 3:会话分段记录

按段展开,每段显示:

- 段号(第 N 段)
- 起止时间
- 归档原因(动作数触发 / 里程碑触发 / 用户主动 / 任务完成自然结束)
- 接续摘要(可展开查看)
- 自检结果(可展开查看)

### 智能体状态扩展

V0.2.2 启用后,执行阶段智能体(前端、后端、测试)的状态行下方增加一行小字:

```
后端智能体  工作中 · 执行任务-2
            第 2 段会话 · 接续自第 1 段
```

不显示 token 百分比。

### 事件流新增条目

V0.2.2 启用后,会话切换在事件流中产生 2 条新事件:

- "{智能体名} · 任务-{N} · 上下文已归档,启动新会话继续"
- "{智能体名} · 任务-{N} · 接续会话已加载任务进度,继续执行"

### 任务卡片不变

任务卡片(出现在任务板)的字段在 V0.2.2 阶段不变,**不显示会话信息**。

理由:任务卡片是高密度信息列表,会话信息属于"展开后的细节",不属于"列表概要"。

## V0.2.2 演示的完整切换样例(任务-2 第 1 段归档)

为了让代码 Agent 实现时有具体参照,本节给出一个完整切换样例。

### 切换前(任务-2 第 1 段会话末尾)

智能体已执行动作:

- 读 src/api/schema/tags.ts
- 设计 GET /api/tags 接口
- 写 GET /api/tags 实现
- 写 GET /api/tags 单元测试
- 设计 POST /api/tags 接口
- 写 POST /api/tags 实现
- 写 POST /api/tags 单元测试
- 跑单元测试,GET 通过、POST 通过
- 设计 PUT /api/tags 接口结构
- 评估 DELETE /api/tags 删除策略(软删除 vs 硬删除)

动作数到达 10,触发归档(规则:动作数 ≥ 10 触发,本数字仅为样例)。

### 接续摘要(本段会话产出)

已完成:

- GET /api/tags 接口实现 + 单元测试通过
- POST /api/tags 接口实现 + 单元测试通过
- PUT /api/tags 接口结构设计完成

决策:

- 标签列表使用游标分页,不用 offset 分页(理由:大标签量下 offset 性能差)
- POST 接口允许重复标签名提交,但内部去重(理由:用户体验优先,后端兼容容错)
- 标签长度限制 20 字符(理由:与 schema 设计的 VARCHAR(20) 对齐)

未决:

- PUT 接口是否支持批量更新 — 留待用户审查时确认
- DELETE 接口是否需要软删除 — 任务-1 schema 已支持软删除,但 API 是否暴露未确认

下一步:

- 完成 PUT /api/tags 实现
- 实现 DELETE /api/tags(默认硬删除,留 PUT 处理软删除场景)
- 写 PUT、DELETE 的单元测试
- 写参数校验逻辑

关键文件:

- src/api/schema/tags.ts(已读)
- src/api/tags/index.ts(已写部分)
- src/api/tags/tags.test.ts(已写部分)

### 自检结果

未完成事项检查:

- 已识别 4 项下一步事项,已记录到"下一步"
- 已识别 2 项未决事项,已记录到"未决"

持久化检查:

- 已生成接续摘要(412 字)
- 已记录 3 条关键决策到决策日志
- 已快照当前 git 工作区(branch: feat/tags-api)

接续准备度:

- 下一段会话基于摘要可以继续 PUT/DELETE 实现,不需要重新读已完成的 GET/POST 代码

### 接续会话(第 2 段)启动

读取:

- 任务-2 任务卡(任务名、目标、依赖、验收标准)
- 第 1 段接续摘要
- 关键文件:src/api/schema/tags.ts、src/api/tags/index.ts、src/api/tags/tags.test.ts(按需读)

不读取:

- 第 1 段会话的完整对话历史
- 第 1 段已读过的其他文件

具体动作数和摘要文字以本节为参照,后续 docs/06 样例数据中保持一致。

## 异常处理

### 异常一:接续摘要不完整

样例:智能体归档时漏写"决策"字段。

V0.2 演示规则:任务详情页"会话分段记录"中该段标注"摘要不完整 · 待补充"红色提示。代码 Agent 实现时不需要真实检测,使用预设异常样例展示即可。

### 异常二:接续会话发现进度回退

样例:第 2 段会话读了接续摘要后,发现摘要里写"GET 已通过测试",但实际工作区代码 GET 接口缺失(智能体可能错记)。

V0.2 演示规则:任务详情页出现"接续不一致 · 需要人工核查"提示,任务状态进入"卡住"标签。代码 Agent 实现时使用预设异常样例展示。

### 异常三:用户主动归档但智能体未完成关键步骤

样例:用户在智能体写到 PUT 一半时主动归档。

V0.2 演示规则:智能体应自动产出"用户主动归档"接续摘要,标注"未完成事项:PUT 接口写到一半 · 未保存"。代码 Agent 实现时使用预设异常样例展示。

## V0.2.0 / V0.2.1 / V0.2.2 批次差异

### V0.2.0 / V0.2.1 阶段

- 不启用任何会话管理规则
- 任务详情页"上下文"Tab 不升级(沿用 V0.1.1)
- 智能体状态行不显示会话信息
- 事件流不出现会话切换事件

### V0.2.2 阶段(本规范完整启用)

- 启用第三章触发规则
- 启用第四章归档规则
- 启用第五章接续规则
- 启用第六章 UI 呈现规则
- 启用第七章样例
- 启用第八章异常处理

## 严禁事项

- 严禁显示伪造的 token 百分比。
- 严禁让接续摘要只是"上一段对话的总结",必须包含决策记录。
- 严禁让新会话读取上一段的完整对话历史。
- 严禁让"会话切换"在 V0.2.0 / V0.2.1 阶段被实现(分批纪律)。
- 严禁把"会话切换"信息加到任务卡片上(它是详情页内容)。
- 严禁让"会话切换"被理解为失败信号,它是正常工作模式的一部分。
