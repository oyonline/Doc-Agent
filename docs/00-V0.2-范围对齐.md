# V0.2 · 范围对齐

## 文档目的

这份文档是 V0.2 的入口,锁定 V0.2 要做什么、不做什么,以及背后的心智模型。后续所有 V0.2 文档(单一事实源、八智能体职责说明、状态机与跳转矩阵、样例数据与事件文案、验收清单、页面任务书)都以本文档为参照展开。在 V0.2 文档骨架完整建立之前,本文档与决策日志共同承担过渡期的范围基准职责。当前唯一事实源会随 V0.2 文档逐步建立,本文档先锁定范围。

## 与 V0.1.1 的关系

V0.2 是 V0.1.1 的"心智重构",不是"功能增量"。

V0.1.1 把"一次智能体执行单个任务"作为主轴,围绕"目标 → 计划 → 任务 → 运行 → 验证 → 审查包 → 人工决策"建立单任务追踪闭环;V0.2 把"一个需求由多智能体协作交付"作为主轴,围绕"需求 → 协同澄清 → 闸门 1 → 三产出并行(PRD‖可行性评估‖任务图)→ 闸门 2 → 多智能体并行执行 → 单任务审查 → 需求级审查 → 需求接受"建立需求交付闭环。两条主轴的差异不是页面数量,而是底层心智:V0.1.1 关心"这一次执行做对了吗",V0.2 关心"这个需求被交付了吗"。

V0.1.1 全部文档已归档至 `docs/archive/v0.1.1/`,仅供历史回溯,不再作为事实源。V0.1.1 已实现的代码资产(全局外壳、设计令牌、技术栈选型,以及任务详情页、执行详情页、单任务审查包页)在 V0.2 中作为"任务级下钻视图"继续保留,不重新实现,但需要重新挂载到 V0.2 的需求-任务-运行三层粒度上。

## V0.2 核心心智

V0.2 主流程(并行版):

```text
需求
 → 协同澄清(产品 + 架构师并行,4 轮)
 → 闸门 1 · 冻结需求,开始规划
 → 三产出并行(PRD ‖ 可行性评估 ‖ 任务图)
 → 闸门 2 · 启动执行(或:小修补 / 重新澄清)
 → 任务池
 → 多智能体并行执行
 → 单任务审查
 → 需求级审查
 → 需求接受
```

异常分支:协同澄清若 ≥ 6 轮仍未对齐,系统自动施加"放弃/拆分"提示,用户决策"放弃 / 拆分 / 继续尝试澄清"。"放弃"或"拆分"将主状态推入终态"已放弃"。

V0.1.1 主流程:

```text
目标 → 计划 → 任务 → 运行 → 验证 → 审查包 → 人工决策
```

主要差异:

- V0.1.1 的"目标"是一句话目标,V0.2 的"需求"是一段完整业务诉求,需经协同澄清(产品 + 架构师并行)对齐后,与可行性评估、任务图一同在规划阶段并行产出。
- V0.1.1 的"计划"是 6 步线性步骤,V0.2 的"任务拆解"产出有依赖关系的任务图,允许多个任务并行。
- V0.1.1 的"任务"近似等价于"一次执行",V0.2 的"任务"是被 PRD 拆解后的子单元,多个任务共同支撑一个需求。
- V0.1.1 只有单任务审查包一层人工决策,V0.2 拆为"单任务审查包(技术 review)"和"需求级审查包(产品验收)"两层。
- V0.1.1 把智能体折叠为 3 类(计划智能体、工作智能体、审查智能体),V0.2 展开为 8 个角色,以匹配真实需求交付中的产品、架构、拆解、调度、执行、测试、审查分工。

## V0.2 解决什么问题

V0.1.1 解决了"如何让单个智能体任务可观察":在一次执行内,用户能清楚看到目标、计划、上下文、日志、改动、验证和审查证据。它把"智能体执行"从黑盒变成可追踪的过程。

V0.2 解决"如何让多个智能体协作完成一个需求时,人类仍能清楚知道每一步、做最终决策":多智能体协作交付时,信息更密集、阶段更多、并行更多,人类很容易被挤出决策环。V0.2 要保证每一次澄清、每一次拆解、每一次执行、每一次审查都能被看到、能被介入,而不是"提一个需求、等一个交付物"。

V0.2 不解决"如何让智能体真正跑起来"。它仍然是高保真静态原型,所有澄清对话、PRD、拆解结果、任务执行进度、审查内容都使用静态样例数据。当前阶段先把多智能体协作的人机交互形态走通,再进入真实实现。

## V0.2 范围

### 包含

- 八个智能体角色:产品、架构师、任务拆解、项目管理、前端、后端、测试、审查。
- 三层粒度:需求 → 任务 → 运行。
- 协同澄清交互:静态演示 4 轮,产品智能体作为主持人,架构师智能体同步在场协同提问。每轮发言归属清晰(产品提业务问题 → 架构师视需要插话补技术问题 → 用户回答),第 4 轮架构师主导提问技术风险与边界。第 4 轮结束后产品智能体整合并展示"澄清结论摘要",由用户在闸门 1 决定是否冻结需求。
- 可行性评估使用风险清单 + 三档评级(高/中/低),不使用百分比。
- 任务图可视化:用自写 SVG 节点 + 连线展示任务依赖与并行关系。
- 任务池调度演示:静态展示项目管理智能体如何按依赖图分发任务到不同执行智能体。
- 两层审查包:单任务审查包(技术 review)+ 需求级审查包(产品验收)。
- 跨任务一致性检查:需求级审查包中由审查智能体提供静态问题清单。
- 端到端样例需求:"新增用户标签功能"(详见样例数据文档)。
- 中文文案。
- 默认严格安全模式。
- 静态样例数据。
- 集成测试作为独立任务存在,与端到端测试区分(集成测试在前后端任务完成后跑,端到端测试在所有任务完成后跑)。
- 任务图含两类依赖:实线箭头表示强依赖(必须等前置完成),虚线表示契约依赖(可并行,但有数据契约关联)。
- 闸门(人工介入)清单(本次升级新增规划阶段 2 闸门,与任务级、需求级闸门统一编号):

  V0.2.0 共 5 个闸门:

  - 闸门 1:冻结需求(协同澄清完成,新增)
  - 闸门 2:启动执行(规划阶段三产出完成,新增)
  - 闸门 3:任务-1 完成(数据层)
  - 闸门 4:任务-6 完成(端到端)
  - 闸门 5:需求级审查(产品验收)

  V0.2.1 共 6 个闸门:

  - 闸门 1:冻结需求(协同澄清完成)
  - 闸门 2:启动执行(规划阶段三产出完成)
  - 闸门 3:任务-1 完成(数据层)
  - 闸门 4:任务-7 完成(集成层)
  - 闸门 5:任务-6 完成(端到端)
  - 闸门 6:需求级审查(产品验收)

  V0.2.2 不新增闸门,仅 Context 切换演示。

- 单任务自动放行机制:其他任务完成后自动进入下游调度,但单任务审查包仍生成,可主动查看。
- 任务卡片"验证结果"字段展示分项数量(如"4/4 项通过"),悬浮或展开任务详情页可看分项详情。
- 任务自带"任务内验证"(单元测试、代码检查、类型检查、契约校验),不触发独立测试任务。
- Context 切换演示:演示智能体在执行中因上下文接近上限而归档新开会话,保证任务连续性与质量。
- 任务详情页"上下文"Tab 升级为"决策与上下文"Tab,展示初始上下文、决策日志、会话切换记录。
- 接续会话强制包含"决策记录"而非仅"动作记录",保证质量不因会话切换降级。
- 智能体状态展示当前会话信息(第几段会话、是否接续),但不显示伪造的 token 百分比。

### 不包含

- 真实知识沉淀。
- 真实并发资源管理。
- 真实代码索引、git worktree、合并队列。
- 真实智能体调用、真实模型调用。
- 真实后端、数据库、模型 API 调用。
- 登录、权限、计费、多用户、移动端。
- V0.3 及之后的能力。
- 真实 token 百分比展示(V0.2 不伪造数字,只展示分段与接续关系)。
- 真实 context 阈值监控逻辑(V0.2 用预设的样例切换点演示)。
- 真实跨任务数据一致性自动检测(V0.2 由审查智能体提供静态问题清单)。
- 任务执行中的实时资源监控、并发限流、智能体负载均衡。

## 八个智能体角色

| 智能体 | 阶段 | 职责 |
|---|---|---|
| 产品智能体 | 规划 | 主持协同澄清、整合澄清结论摘要、并行产出 PRD |
| 架构师智能体 | 规划 | 协同澄清提技术问题(第 4 轮主导)、并行产出可行性评估、识别风险 |
| 任务拆解智能体 | 规划 | 把 PRD 拆成有依赖关系的任务图 |
| 项目管理智能体 | 执行 | 按任务依赖图调度任务、监控并行执行 |
| 前端智能体 | 执行 | 执行前端类任务 |
| 后端智能体 | 执行 | 执行后端类任务 |
| 测试智能体 | 执行 | 执行测试类任务 |
| 审查智能体 | 审查 | 复审单任务结果、做需求级跨任务一致性检查 |

## 三层粒度

| 层级 | 单元 | 状态机 |
|---|---|---|
| 需求 | 一个完整业务诉求(如"新增用户标签功能") | 待澄清 → 规划中 → 执行中 → 待审查 → 已完成;终态分支:已放弃(放弃 / 拆分) |
| 任务 | 被拆解后的执行单元(如"标签 CRUD API") | 待处理 → 执行中 → 验证中 → 待审查 → 已完成 |
| 运行 | 任务的单次执行记录 | 未开始 → 进行中 → 已完成 → 已失败 → 已停止 |

异常标签作为标签叠加,不作为主状态。"已放弃"是除"已完成"之外的另一终态主状态,带子语义"放弃 / 拆分"。

"规划中"主状态在 UI 上以**轻量进度指示**呈现 3 段子阶段:**澄清 ▶ 三产出 ▶ review**。状态色块保持单一(规划中色),状态文案保持"规划中",状态色块旁加 3 段进度指示,当前阶段高亮、已走过阶段保留高亮、未达阶段弱化。需求工作台行内紧凑展示时仅显示状态色块 + "规划中" 文案,不展示进度指示;需求详情页顶部、任务板需求分组标题展示完整进度指示。具体视觉规范由 `docs/08 §页面通用规范` 落实。

## V0.2 分批实现计划

V0.2 范围内的设计概念全部记录在文档中,但实现按以下三批顺序展开,确保每批可独立 checkpoint。

### V0.2.0 · MVP 基础

包含:

- 八智能体角色完整呈现(右侧状态栏分组展示)。
- 三阶段并行调度(规划 → 执行 → 审查)。
- 单任务审查包(沿用 V0.1.1 资产)+ 需求级审查包(新增)。
- 协同澄清交互(4 轮静态演示,产品 + 架构师并行)。
- 闸门 1(冻结需求,开始规划)+ 闸门 2(启动执行)交互完整,含小修补(每段产出右上角)、重新澄清、放弃 / 拆分等异常分支按钮。
- 任务图基础版(节点 + 实线箭头,不含虚线契约依赖)。
- 样例需求使用统一的 7 任务数据,V0.2.0 阶段任务-7 在任务图中作为"样例占位"展示(节点存在但标注"待 V0.2.1 激活"),不参与调度演示与人工介入。
- 闸门完整 5 个:闸门 1(冻结需求)、闸门 2(启动执行)、闸门 3(任务-1 后)、闸门 4(任务-6 后)、闸门 5(需求级审查)。
- 任务-2、3 完成后直接放行任务-4、5(V0.2.0 阶段不经任务-7 闸门)。

### 文档与实现关系

V0.2 全部设计概念在文档骨架建立时一次性写入,实现按 0/1/2 顺序推进。每批实现完成后做 checkpoint,通过后进入下一批。

任何批次内发现需要回退或修改前置批次的范围,必须先停下记录到决策日志,不允许跳批改动。

## V0.2 不做什么(强约束,严禁回退)

- 严禁把 V0.2 退回成"单任务执行追踪"。
- 严禁让首页直接是 textarea 输入目标。
- 严禁让"一个需求 = 一个任务"的语义出现。
- 严禁让八个智能体折叠回三个智能体。
- 严禁恢复 V0.1.1 的"目标 → 计划 → 任务"心智作为主轴。
- 严禁照搬 Gas Town 的命名体系。
- 严禁让"任务卡片"展示伪造的 token 百分比。
- 严禁让"集成测试"和"端到端测试"被合并成一个测试任务。
- 严禁让 V0.2.0 实现包含 V0.2.1 或 V0.2.2 的功能(分批纪律)。
- 严禁让"已驳回"以任何变体形式留在状态机或样例数据中(包括换名留旧逻辑,如改称"待修订"但仍走"PRD 已定 → 二次澄清"动线)。
- 严禁让协作模型从"并行"降级回 V0.2 串行模型或 V0.1.1 的 3 智能体。
- 严禁让协同澄清变成"两个 AI 同时大段说话"的混乱形态——必须有清晰的轮次和发言归属(产品提问 / 架构师插话 / 用户回答)。
- 严禁让协作模型升级触发 V0.2 整体范围扩大(本次只是协作模型调整,不是范围扩展)。
