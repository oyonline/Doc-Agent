## 2026-04-28：产品定位为智能体交付管理工作台

决策：

本项目定位为管理智能体交付的工作台，不是聊天机器人，也不是普通任务管理工具。

原因：

核心价值是管理目标、计划、执行证据、验证结果和审查决策，而不是对话本身。

影响：

界面中心必须是任务板、任务详情页、执行详情页和审查包，不做聊天机器人界面。

## 2026-04-28：左侧导航收敛为五项

影响：

## 2026-04-28：“运行”和“审查”不作为一级导航

决策：

“运行”和“审查”不进入左侧一级导航。

原因：

影响：

## 2026-04-28：审查包必须支持同页差异预览

决策：

审查包页必须支持点击修改文件清单后在同页展示静态差异预览。

原因：

审查包是最终决策入口，用户需要在同一页看到总结、验收标准、改动、验证和风险，不能为了看 diff 跳走。

影响：

低保真原型需要设计同页抽屉或清单下方差异区域；默认选中第一个文件，至少展示一个静态 diff 样例。

## 2026-04-28：拒绝并反馈与重新执行必须区分

决策：

`拒绝并反馈` 与 `重新执行` 是两个不同操作。

原因：

拒绝并反馈会形成新上下文；重新执行则不带新反馈，按原计划再执行一次。

影响：

审查包页底部左侧固定放 `拒绝并反馈`，右侧固定放 `接受结果`，`重新执行` 与 `让审查智能体复查` 放入 `更多操作`。

## 2026-04-29:产品验证观察 · 人肉 Orchestrator 痛点

观察:

在 V0.2 文档骨架建设过程中,项目主导者亲身体感到"人肉 Orchestrator"的痛苦——明明 AI 之间可以传递的信息,必须通过人类复制粘贴中转。每段提示词都需要人类手动从 Claude.ai 复制到 Claude Code 终端,产出再手动复制回来,人在两个 AI 之间充当传递接口。

意义:

这种痛苦正是 V0.2 想要解决的核心痛点的镜像。业务方提需求后,不应该手动协调多个智能体,而是由系统自动调度。当前工具栈限制(Claude.ai 与 Claude Code 之间无 IPC 通道)暂时无法消除人肉中转,但这一观察从产品主导者自身体感中确认了 V0.2 产品方向的真实价值。

后续行动:

后续提示词改为"自驱动模式"——能机械推进的环节,Agent 完成产出后直接进入下一段任务的等待状态,不需要每段都让用户 review 全文。仅在以下情况停下来:文档之间矛盾、提示词内部不一致、产品决策性问题、主动添加超出格式补全范围。

V0.2 / V0.3 锚点:

工具层面消除"人肉 Orchestrator"需要 IPC 通道(Claude.ai SDK 与 Claude Code SDK 互通),记录为产品后续锚点。

## 2026-04-29:工程观察 · Claude Code Write 工具与 Vite HMR 兼容性

观察:

V0.2.0 实现阶段需求工作台页面跑出来后,浏览器始终显示旧版占位内容。诊断发现:磁盘文件已是新版,但 Vite 启动时索引的旧版本一直在内存里,后续 Claude Code 通过 Write 工具的多次写入未触发 chokidar 文件监听通知,Vite 不重新 transform 模块,浏览器拿到的是旧 JS。

根因:

Claude Code 的 Write 工具底层写入路径(可能涉及 atomic rename、临时文件、或 macOS fs hook)绕过了 Vite chokidar 监听。

应对方案:

代码 Agent 在做以下操作时,完成后必须重启 dev server,而非依赖 HMR:

- 大幅重写或新建 .tsx / .ts 文件(>100 行改动)
- 修改路由表 main.tsx
- 修改全局 mock 数据 lib/mock-data.ts
- 修改任何 import 关系

小幅样式调整、文案改动、单行 bug 修复仍可依赖 HMR。

代码 Agent 完成大幅改动并通过 TS 检查后,默认 kill -> pnpm dev 重启,等首次编译完成再告知用户。

V0.3 锚点:

如果后续切换到其他工具栈或 Claude Code 工具机制变更,本约束可能失效,届时重新评估。

## 2026-04-29:产品观察 · 信息形态决定视觉调性

**叙事型页面**(理解过程、跟随推理):
- 需求详情页(澄清 → PRD → 评估 → 任务图 → 进度)
- 执行详情页(时间线 → 日志 → 改动)

这类页面天然包含文档型内容(对话、段落、报告)。纯控制台化会损伤可读性,纯文档化会失去工具感。

叙事型页面的视觉处理原则:

1. 内容主体保持文档型呈现(可读性优先)
2. 顶部/侧边/底部加控制台型"装饰带"(状态条、进度指示器、操作按钮)
3. 关键数据点(数字、ID、时间戳、状态)用控制台元素(等宽数字、状态色块)
4. 不强行把段落改成槽位、不强行把对话改成数据流

V0.2 视觉改造策略:

- 控制台型 5 个页面:统一用控制台风格(字体、字号、颜色、密度、动效)
- 叙事型 2 个页面:保留文档主体 + 注入控制台元素

V0.2.1 / V0.2.2 阶段在改造后视觉系统上推进。

## 2026-04-30:V0.2 协作模型从串行升级为并行(产品架构升级)

### 触发背景

项目主导者在 V0.2.0 第一个完整可走查版本(commit 11ef34b)实际看 UI 后,识别出当前 docs/02 的智能体协作模型为简化的"产品先聊完再叫架构师评估"串行模型,与真实软件团队的"产品 + 技术负责人共同对接业务方"协作不一致,要求升级为并行模型。

升级前(串行):产品独立主持 3 轮澄清 → 用户确认 PRD → 架构师评估,识别高风险时触发"已驳回",回退产品做二次澄清 → 二次评估降级 → 任务拆解。

升级后(并行):
- 阶段 A 协同澄清:产品 + 架构师同时在场,产品主持,架构师视需要插话补技术问题,4 轮对齐 → 产品整合澄清结论摘要 → 用户在闸门 1 冻结需求。
- 阶段 B 三产出并行:产品(PRD)+ 架构师(可行性评估)+ 任务拆解(任务图)三智能体并行启动,任务拆解在前两份接近完成时启动 → 用户在闸门 2 决策(启动执行 / 小修补 / 重新澄清)。
- 阶段 C-E:沿用现有 V0.2 设计,本次不修改。
- 异常分支:协同澄清 ≥ 6 轮未对齐,系统自动施加"放弃/拆分"提示,用户决策"放弃 / 拆分 / 继续尝试澄清"。

### 原因

真实软件团队启动新需求时,产品 + 技术负责人 + tech lead 一起和业务方对齐——"产品先聊完再叫技术 review"是文档简化模型,不符合真实工程协作。

V0.2 的产品定位是"管理智能体交付的工作台",其展示的协作模型应当与真实软件团队工作方式对齐,而不是简化为单线流转。本次升级让 V0.2 的产品架构匹配真实软件团队工作方式,也为 V0.3 引入更多真实工程概念(并行执行、并行审查、多负责人决策)铺平产品语义基础。

### V0.2 / V0.3 锚点

- **V0.2 范围内本次解决**:规划阶段(澄清 + 三产出)的并行化、闸门 1 / 闸门 2 / 小修补 / 重新澄清 / 放弃 / 拆分异常分支的完整状态机与按钮触发矩阵、6 主状态升级、规划中子阶段进度指示。
- **V0.3 锚点**:
  1. 执行阶段是否也需要"产品 + 架构师 + tech lead"层级的多负责人介入(当前任务级闸门只有用户单向决策,无 Agent 协同)。
  2. 审查阶段是否需要"产品负责人 + 测试负责人"并列介入(当前需求级审查只是单一审查智能体)。
  3. 接受阶段是否需要"业务方 + 产品 + 工程"分别签字(当前只有"接受需求结果"单按钮)。
  4. 多需求并行时,产品智能体与架构师智能体的"在场"是按需求独占还是按时间片复用(当前模型假设单需求,未涉及调度)。
  5. 协同澄清的轮次阈值(N=4 / 触发"放弃/拆分"=6)是否在多需求 / 复杂需求场景下需要动态调整。
