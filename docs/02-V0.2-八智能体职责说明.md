# V0.2 · 八智能体职责说明

## 文档目的

本文档定义 V0.2 八个智能体各自的职责边界、输入输出、状态细分与协作关系,作为后续 V0.2 文档(状态机与跳转矩阵、任务调度规范、Context 管理规范、样例数据与事件文案、验收清单、页面任务书)在涉及智能体时的引用基准。本文档不定义页面、不定义跳转、不定义具体样例数据;这三类内容由后续对应文档承担。本文档亦不重复 `docs/00-V0.2-范围对齐.md` 中已锁定的整体范围与分批计划,只在涉及单个智能体的演示规范时引用对应批次。

## 三阶段分组

V0.2 的八个智能体按交付流程分为三个阶段:

| 阶段 | 智能体 | 阶段产出 |
|---|---|---|
| 规划阶段 | 产品智能体、架构师智能体、任务拆解智能体 | PRD、可行性评估、任务图 |
| 执行阶段 | 项目管理智能体、前端智能体、后端智能体、测试智能体 | 任务执行结果、单任务审查包 |
| 审查阶段 | 审查智能体 | 单任务复审、需求级跨任务一致性检查 |

注:项目管理智能体作为执行阶段的"调度者",不直接产出代码或测试结果,但它的工作贯穿整个执行阶段。

## 智能体通用规范

所有八个智能体共享以下规范,后续单独章节不再重复:

- 每个智能体只在自己的阶段工作,不允许跨阶段执行(产品智能体不写代码、测试智能体不做架构评估)。
- 每个智能体的输入输出必须明确定义,不允许"任意输入、任意输出"的模糊接口。
- 每个智能体的状态只有两种主状态:空闲、工作中。工作中可叠加细分状态文案(如"澄清中"、"撰写 PRD 中")。
- V0.2 阶段所有智能体使用静态样例数据演示,不真实调用模型。
- 智能体之间不直接通信,所有协作通过文档(PRD、任务图、审查包)和调度器(项目管理智能体)中介。
- 执行阶段的智能体(前端、后端、测试)在 V0.2.2 启用 Context 管理深度后,支持会话切换;切换时必须产出包含决策记录的接续摘要。
- 智能体的工作产出必须可被人工查看(产品 PRD、任务图、单任务审查包、需求级审查包),不允许"黑盒产出"。

## 八个智能体详细说明

每个智能体使用统一格式,严格按以下子结构展开:

- **阶段**
- **核心职责**
- **输入**
- **输出**
- **协作关系**
- **典型动作**
- **状态细分**(空闲 / 工作中可叠加的细分文案)
- **V0.2 演示规范**(指定本智能体在 V0.2.0 / V0.2.1 / V0.2.2 各批次的演示要点)

按以下顺序写八个智能体:

### 4.1 产品智能体

- **阶段**:规划
- **核心职责**:接收用户原始需求,通过多轮澄清把模糊需求变成清晰可执行的 PRD。
- **输入**:用户提交的原始需求(自然语言描述)。
- **输出**:多轮澄清记录、最终 PRD(含背景、目标、范围、用户故事、需求级验收标准)。
- **协作关系**:产出的 PRD 提交给架构师智能体做技术可行性评估;不直接与执行阶段智能体交互;遇到架构师评估出的高风险时可被触发二次澄清。
- **典型动作**:提问澄清、归纳用户回答、整理需求边界、生成 PRD 文档。
- **状态细分**:空闲 / 工作中(澄清中 · 第 N 轮、撰写 PRD 中)。
- **V0.2 演示规范**:多轮澄清固定演示 3 轮(产品提问 → 用户答 → 产品再问 → 用户答 → 产品确认 → 用户确认进入 PRD)。

### 4.2 架构师智能体

- **阶段**:规划
- **核心职责**:基于 PRD 做技术可行性评估,识别风险,给出实现方向建议(不写代码)。
- **输入**:产品智能体产出的 PRD。
- **输出**:可行性评估报告(含风险清单 + 三档评级)、实现方向建议、关键技术决策记录。
- **协作关系**:评估结果与 PRD 一起提交给任务拆解智能体;遇到高风险时可触发产品智能体二次澄清。
- **典型动作**:扫描代码库、识别影响范围、列出技术风险、给出实现方案选项。
- **状态细分**:空闲 / 工作中(可行性评估中、风险识别中)。
- **V0.2 演示规范**:可行性评估使用风险清单 + 三档评级(高/中/低),不使用百分比。每个风险条目包含描述、影响范围、缓解建议三个字段。

### 4.3 任务拆解智能体

- **阶段**:规划
- **核心职责**:把 PRD 和实现方向拆解成有依赖关系的任务图,标注前后端测试归属、并行可能性。
- **输入**:PRD + 架构师的可行性评估与实现方向。
- **输出**:任务图(节点为单个任务,边为依赖关系),每个任务含:任务名、负责智能体类型、输入契约、输出契约、验收标准、依赖前置任务。
- **协作关系**:任务图提交给项目管理智能体进入任务池调度。
- **典型动作**:识别任务边界、判断依赖、判断并行可能性、定义验收标准。
- **状态细分**:空闲 / 工作中(拆解中、契约定义中)。
- **V0.2 演示规范**:任务图用 SVG 节点 + 连线展示。V0.2.0 阶段只产出实线箭头(强依赖);V0.2.1 阶段补充虚线契约依赖。样例需求拆出 7 个任务(详见样例数据文档)。

### 4.4 项目管理智能体

- **阶段**:执行
- **核心职责**:按任务依赖图调度任务,把可调度任务分发到对应执行智能体,监控执行状态。
- **输入**:任务拆解智能体产出的任务图。
- **输出**:任务执行调度记录(哪个任务何时分发给哪个智能体、何时完成)、执行阶段进度。
- **协作关系**:与前端、后端、测试智能体直接交互;不直接执行代码,只做调度。
- **典型动作**:从任务池取任务、判断依赖是否满足、分发任务、收集结果、推进任务图状态。
- **状态细分**:空闲 / 工作中(调度中 · X/N 任务进行,已完成 Y)。
- **V0.2 演示规范**:任务池视图静态展示项目管理智能体如何按依赖图调度。

**调度规则**(V0.2 演示)

1. 任务的所有实线前置依赖全部到达"已完成"状态时,任务变成"可调度"。
2. 多个"可调度"任务同时进入分发队列,不排队、不一个一个分发。
3. 当一个执行智能体(如前端智能体)同时收到多个可调度任务时,V0.2 假设它可以同时处理。
4. 任务完成后默认自动放行下游任务进入调度;在指定的人工介入节点(任务-1 后、任务-7 后、任务-6 后)暂停调度,等待人工接受单任务审查包后才推进。
5. 单任务审查包始终生成,可主动查看,但不阻塞默认自动放行流程。

### 4.5 前端智能体

- **阶段**:执行
- **核心职责**:执行前端类任务(组件、页面、交互、样式)。
- **输入**:项目管理智能体分发的前端任务(含输入契约、输出契约、验收标准)。
- **输出**:前端代码改动、任务内验证结果、单任务审查包。
- **协作关系**:由项目管理智能体分发任务,完成后回报给项目管理智能体;改动可能依赖后端智能体的接口产出(通过任务图依赖关系约束)。
- **典型动作**:阅读现有组件、写代码、跑任务内验证(单元测试、代码检查、类型检查、契约校验)、生成审查包。
- **状态细分**:空闲 / 工作中(执行任务-X、跑任务内验证中、生成审查包中)。
- **V0.2 演示规范**:V0.2 不真实生成代码,样例数据中前端任务展示静态 diff。任务内验证以"X/N 项通过"形式在任务卡片展示,详情可在任务详情页查看。V0.2.2 启用后支持会话切换。

### 4.6 后端智能体

- **阶段**:执行
- **核心职责**:执行后端类任务(API、数据库、业务逻辑)。
- **输入**:项目管理智能体分发的后端任务。
- **输出**:后端代码改动、API 契约更新、数据库迁移、任务内验证结果、单任务审查包。
- **协作关系**:由项目管理智能体分发任务;前端智能体可能依赖其 API 产出。
- **典型动作**:设计 schema、写 API、跑任务内验证、生成审查包。
- **状态细分**:空闲 / 工作中(执行任务-X、跑任务内验证中、生成审查包中)。
- **V0.2 演示规范**:同前端智能体,V0.2 用静态样例数据。V0.2.2 启用后支持会话切换。

### 4.7 测试智能体

- **阶段**:执行
- **核心职责**:执行测试类任务(集成测试、端到端测试)。任务内单元测试由前后端智能体自带,不归测试智能体。
- **输入**:项目管理智能体分发的测试任务,通常依赖前后端任务完成。
- **输出**:测试用例、测试报告、单任务审查包。
- **协作关系**:由项目管理智能体分发任务;依赖前后端智能体的产出作为被测对象。
- **典型动作**:写测试用例、跑测试、生成报告。
- **状态细分**:空闲 / 工作中(写用例中、跑测试中、生成报告中)。
- **V0.2 演示规范**:测试智能体在 V0.2.0 只承担端到端测试(任务-6);V0.2.1 启用后追加集成测试(任务-7)。

### 4.8 审查智能体

- **阶段**:审查
- **核心职责**:复审单任务结果,做需求级跨任务一致性检查。
- **输入**:单任务审查包(技术 review)、需求级所有任务汇总(产品验收前)。
- **输出**:单任务复审结论、需求级跨任务一致性问题清单。
- **协作关系**:在两个时机被调用——单任务待审查时(可被人工触发"让审查智能体复查")、需求级审查包生成前(自动生成跨任务问题清单)。
- **典型动作**:对比 PRD 与实现、检查 API 契约对齐、检查测试覆盖、识别跨任务问题。
- **状态细分**:空闲 / 工作中(复审任务-X 中、跨任务一致性检查中)。
- **V0.2 演示规范**:需求级审查包中由审查智能体提供静态问题清单,问题清单按"严重 / 警告 / 提示"三档分级。

## 智能体协作流程图(文字版)

```text
用户提交需求
   │
   ▼
产品智能体 ── 多轮澄清(3 轮) ──→ PRD
                                │
                                ▼
                        架构师智能体 ── 可行性评估 + 风险清单
                                │
                                ▼
                        任务拆解智能体 ── 任务图
                                │
                                ▼
                        项目管理智能体 ── 任务池调度
                                │
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
    前端智能体              后端智能体               测试智能体
   (任务-4、5)           (任务-1、2、3)           (任务-7、6)
        │                       │                       │
        └────── 单任务审查包 ───┴───── 单任务审查包 ───┘
                                │
                                ▼
                      (人工介入 4 次:任务-1、7、6 后,
                       以及最终需求级审查)
                                │
                                ▼
                        审查智能体 ── 跨任务一致性检查
                                │
                                ▼
                          需求级审查包
                                │
                                ▼
                          人工最终决策
```

## 状态展示规范

右侧状态栏八个智能体按三阶段分组展示:

```
规划阶段
  产品智能体     [空闲 / 工作中(澄清中 · 第 N 轮、撰写 PRD 中)]
  架构师智能体   [空闲 / 工作中(可行性评估中、风险识别中)]
  任务拆解智能体 [空闲 / 工作中(拆解中、契约定义中)]

执行阶段
  项目管理智能体 [空闲 / 工作中(调度中 · X/N 任务进行,已完成 Y)]
  前端智能体     [空闲 / 工作中(执行任务-X、跑任务内验证中、生成审查包中)]
  后端智能体     [空闲 / 工作中(执行任务-X、跑任务内验证中、生成审查包中)]
  测试智能体     [空闲 / 工作中(写用例中、跑测试中、生成报告中)]

审查阶段
  审查智能体     [空闲 / 工作中(复审任务-X 中、跨任务一致性检查中)]
```

每个分组之间留 16px 间距,分组小标题使用 text-xs 弱化色。智能体当前会话信息(V0.2.2 启用后)以小字附在状态行下方,不显示百分比。

## 严禁事项

- 严禁把八个智能体折叠回 V0.1.1 的三个智能体(计划、工作、审查)。
- 严禁让一个智能体跨阶段执行(产品智能体不写代码、测试智能体不做架构评估、审查智能体不做拆解)。
- 严禁出现"通用智能体""万能智能体"等模糊角色。
- 严禁照搬 Gas Town 命名体系(Mayor、Witness、Beads 等)。
- 严禁使用英文角色名(如 Product Agent),统一中文。
- 严禁让前后端智能体的"任务内验证"被独立任务化。
- 严禁让测试智能体承担单元测试(单元测试归任务自带)。
- 严禁绕过项目管理智能体让执行阶段智能体直接通信。
